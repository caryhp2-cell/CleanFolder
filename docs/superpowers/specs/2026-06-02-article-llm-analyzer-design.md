# Article LLM Analyzer via Phi-3.5 Mini ONNX DirectML

## Goal

Improve the Article Analysis feature by replacing the current rule-based extractive analyzer with a local generative LLM analyzer.

The new analyzer uses `microsoft/Phi-3.5-mini-instruct-onnx` and runs through ONNX Runtime GenAI with the Windows DirectML GPU route. The priority is a better user experience: higher-quality article summaries, suggested titles, and key sentence selection while keeping analysis local.

## Approved Direction

- Use Phi-3.5 Mini Instruct ONNX for Article Analysis.
- Prefer the INT4 GPU / DirectML model variant.
- Do not keep the old extractive analyzer as a fallback.
- Do not treat CPU, GPU, and NPU as fallback tiers.
- Select the best route for this use case: GPU through DirectML.
- If the model route is unavailable or output validation fails, show a clear error instead of silently using another analyzer.
- Keep the existing rename/OCR path unchanged for now.

## Why DirectML GPU

Phi-3.5 Mini ONNX is published with optimized CPU and GPU variants and is intended for ONNX Runtime generation. On Windows, DirectML provides broad GPU support across common DirectX 12-capable hardware. For this feature, DirectML is the best default because article generation benefits from GPU acceleration and should feel quicker than CPU generation.

The existing OpenVINO NPU policy remains useful for the current OCR and compact ONNX routes, but it is not the best assumption for this decoder LLM generation path.

References:

- `microsoft/Phi-3.5-mini-instruct-onnx`: https://huggingface.co/microsoft/Phi-3.5-mini-instruct-onnx
- ONNX Runtime DirectML Execution Provider: https://onnxruntime.ai/docs/execution-providers/DirectML-ExecutionProvider.html
- ONNX Runtime GenAI Phi tutorial: https://onnxruntime.ai/docs/genai/tutorials/phi3-python.html

## User Flow

1. User opens the Article Analysis tab.
2. User chooses a `.txt`, `.md`, `.pdf`, or `.docx` article.
3. The app extracts local article text using the existing document extraction code.
4. The app validates that the Phi article model assets are present and usable.
5. The app verifies the configured article inference route is DirectML GPU.
6. The app prompts Phi to produce structured article analysis JSON.
7. The app validates the JSON.
8. If validation succeeds, the UI shows the title, summary, key sentences, and reason.
9. If validation fails, the UI shows an error explaining the failed condition.

## Model Assets

Add a new article model entry to `models/manifest.json`.

The entry should identify the model as an article LLM route, not as a document title embedding model. The committed manifest must contain real SHA-256 hashes generated by `scripts/fetch_models.py`; the app should never rely on illustrative hash strings.

```json
{
  "id": "article-llm-phi35-mini-directml-v1",
  "path": "phi-3.5-mini-instruct-onnx/gpu/gpu-int4-awq-block-128",
  "task": "article-llm-analysis",
  "required_provider": "DmlExecutionProvider",
  "source_repo": "microsoft/Phi-3.5-mini-instruct-onnx",
  "source_revision": "main",
  "source_subdir": "gpu/gpu-int4-awq-block-128",
  "required_files": [
    "chat_template.jinja",
    "config.json",
    "configuration_phi3.py",
    "genai_config.json",
    "model.onnx",
    "model.onnx.data",
    "special_tokens_map.json",
    "tokenizer.json",
    "tokenizer_config.json"
  ]
}
```

Phi ONNX assets are directory-based rather than a single small ONNX file, so the implementation should support validating a model directory. Directory validation should use a manifest-owned file list with one SHA-256 hash per required file. The JSON above shows the required file names; the fetch script writes the final hash values.

## Runtime

Add a small Article LLM runtime module responsible for:

- locating the article model asset directory
- constructing an ONNX Runtime GenAI model
- configuring DirectML execution
- formatting the prompt
- running generation
- parsing JSON output

The runtime should be lazy-loaded. The app should not load the large model at startup. It should load only when Article Analysis is requested.

## Prompt Contract

The prompt should ask the model to return JSON only:

```json
{
  "suggested_title": "short article title",
  "summary": "3 to 5 sentence summary",
  "key_sentences": [
    "important sentence copied from the article",
    "important sentence copied from the article"
  ],
  "reason": "brief explanation of why these points matter"
}
```

The prompt should tell the model:

- do not invent facts
- keep the title short
- write the summary from the article only
- copy key sentences from the article text
- return JSON without Markdown fences

## Output Validation

The analyzer accepts model output only when:

- output is valid JSON
- `suggested_title` is a non-empty string
- `summary` is a non-empty string
- `key_sentences` is a non-empty list of strings
- each key sentence has a close match in the source text
- `reason` is a string

If validation fails, the result status is `ERROR`. The app does not fall back to extractive analysis.

## Error Handling

Article Analysis should return clear errors for:

- missing Phi model assets
- invalid model manifest
- DirectML provider unavailable
- model load failure
- generation failure
- invalid JSON output
- hallucinated or unmatched key sentences
- empty article text

Errors should be shown in the Article Analysis UI without crashing the app.

## Testing Strategy

Testing should be intentionally lightweight to shorten development time and reduce human intervention.

Routine tests should not load the real Phi model.

Required automated tests:

- fake generator output is converted into an `ArticleAnalysisResult`
- invalid JSON returns `ERROR`
- hallucinated key sentences return `ERROR`
- empty article text returns `ERROR`
- manifest validation accepts directory-based article model assets
- article runtime is lazy-loaded rather than imported at app startup

Optional real-model smoke test:

- run only when an environment variable such as `RUN_REAL_ARTICLE_LLM=1` is set
- skip by default in normal `pytest`
- verify that the real DirectML model can load and produce parseable JSON

This keeps the default development loop fast while still leaving a direct way to validate the real model route.

## Implementation Boundaries

In scope:

- Article Analysis model route
- model manifest support for directory-based assets
- DirectML availability check for Article Analysis
- prompt, generation, JSON parsing, and validation
- Article Analysis UI error display updates
- README updates explaining the GPU DirectML article route

Out of scope:

- changing rename suggestions
- changing OCR
- changing PDF/image OCR NPU policy
- adding CPU fallback
- adding cloud model calls
- keeping the old extractive analyzer as fallback

## Open Decisions Resolved

- Model: Phi-3.5 Mini Instruct ONNX.
- Runtime target: DirectML GPU.
- Fallback: none.
- Test scope: minimal fake-generator tests plus optional real-model smoke.
- User review: keep a single review checkpoint before implementation, then proceed with the smallest useful implementation plan.
